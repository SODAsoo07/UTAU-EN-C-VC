# EN C+VC (C+Bridge) Format Overview

## 1) 목적
이 형식은 영어(GenAm 기준)에서 `CV` 단독형보다 자연스러운 연결을 유지하면서도, 정통 VCCV보다 녹음량을 줄이기 위한 절충안입니다.

- 핵심: `V + -C/C + VC + (대표)CC` 조합
- 전략: 자음을 독립 단위로 재사용하고, 실제 자연스러움에 큰 영향을 주는 구간은 `VC/CC`로 보강

## 2) 녹음 블록 구조
현재 `MIN/MAX` 공통 구조는 아래 6개 블록입니다.

1. `V only`: 모음 핵심 음색
2. `-C`: 어두 자음(phrase head) 추출용
3. `C medial helper`: 어중 자음 보정/대체용
4. `VC anchors`: 모음→자음 연결 핵심
5. `CC onset`: 자주 쓰는 자음군 시작부
6. `CC coda`: 자음군 종성/말미 연결

## 3) 이 형식이 실제로 유리한 점
- `CV` 계열보다 자음 연결(특히 어중/어말)이 자연스럽습니다.
- 고빈도 `VC`를 별도로 잡아 발음 명료도와 리듬 안정성이 올라갑니다.
- `CC`를 대표군으로 제한해서 녹음량 폭증을 억제할 수 있습니다.
- 커스텀 포네마이저와 결합 시, 누락 alias를 단계적으로 fallback 할 수 있습니다.

## 4) 품질과 녹음량의 현실적 트레이드오프
- `MIN`: 적은 녹음량, 빠른 제작/수정. 단, 드문 자음군에서 fallback 의존도가 큼.
- `MAX`: 녹음량 증가, 하지만 복잡한 cluster/말미에서 자연스러움이 안정적.

결론적으로:
- 빠른 배포/프로토타이핑: `MIN`
- 합성 안정성과 고음질 우선: `MAX`

## 5) 현재 포네마이저 호환 관점의 alias 계층
커스텀 포네마이저(EN C+VC)는 대략 아래 순서로 alias를 찾습니다.

- 모음: `V`, 필요 시 `VV`
- 어두 자음: `-C`, `-C(공백 없는 변형)`, `_C`, `C`
- 어중 자음군: `CC`(없으면 단일 C fallback)
- 어말 연결: `VC`, `VC-`, 이후 `CC/CC-`

즉, 음질은 `VC/CC` 품질이 좌우하고, 사용성은 `-C/C` 품질이 좌우합니다.

## 6) 실사용 권장 운영 방식
1. `MIN`으로 먼저 은행을 완성합니다.
2. 실제 곡 렌더 후 깨지는 구간만 `MAX` 또는 보강 리스트로 추가합니다.
3. `oto`는 먼저 유형별 기본값으로 배치하고, 문제 alias만 예외 튜닝합니다.
4. 포네마이저 fallback 로그(또는 누락 alias)를 기준으로 녹음 보강 우선순위를 정합니다.
